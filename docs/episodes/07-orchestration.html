07 - Orchestration, Docker, and logs

How federations are actuated from a script and what to inspect when something fails.

Profiles Flower

Profiles connect names like region-eu or global to each SuperLink endpoint.

Run config

The orchestrator builds TOML strings for --run-config. Each run receives the level, region, initial checkpoint, output checkpoint, and weights.

What to notice
  • Lines 70-80 serialize Python values into Flower's run-config syntax.
  • Lines 83-95 build and optionally execute the flwr run . <profile> --stream --run-config ... command.
  • Lines 98-133 define the regional run config: level, region, previous global checkpoint, output regional checkpoint, and regional sample count.
  • Lines 136-166 define the global run config: previous global checkpoint, output global checkpoint, and total regional sample count.

Hierarchical loop

What to notice
  • Lines 208-228 run each regional federation first. These runs create the regional checkpoints consumed by the gateways.
  • Lines 230-243 run the global federation after all regional checkpoints for that round exist.
  • Line 245 prints the latest global checkpoint path, which is what evaluation should load next.
  • The order is intentional: regional training happens before global aggregation in every outer round.

Diagnostic commands

What progress looks like

When make demo runs, the orchestrator prints each flwr run command before executing it. Flower then streams server/client progress: sampled nodes, train/evaluate messages, completed rounds, checkpoint writes, evaluation reports, and finally a compact metrics CSV.
