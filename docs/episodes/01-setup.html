01 - Setup from scratch

How to prepare Python, dependencies, Kaggle credentials, and the main targets.

Prerequisites

  • Python 3.11 or another version compatible with Flower 1.29.
  • Docker Desktop or Docker Engine with Compose.
  • Kaggle CLI credentials stored outside the repository.
  • GitHub CLI only if you want to publish or configure Pages from the terminal.

Installation

Credentials

Do not put real tokens in Git. Use environment variables or ~/.kaggle/kaggle.json. The repository ignores data, checkpoints, and common credential files.

Main targets

What to notice
  • Lines 20-25 separate download/cleaning from partitioning, so data preparation can be rerun without rebuilding Docker.
  • Lines 39-42 call the hierarchical orchestrator for normal training runs.
  • Lines 54-58 define the compact local demo and delegate the visible transcript to scripts/run_demo.py.
  • Lines 60-70 keep the Docker route available through make demo-docker without making it the default live demo path.
  • Lines 75-77 execute the educational notebooks and rebuild GitHub Pages without retraining.
  • The demo target assumes the dataset and partitions already exist. This avoids downloading a large dataset every time someone wants to show the algorithm running.

Direct demo

Once data and partitions exist, make demo starts a local Flower Deployment Runtime with real SuperLinks, SuperNodes, and SuperExec processes. It runs one global round with one regional round, evaluates, predicts, executes the notebook vignettes, and refreshes the renders.

Package dependencies

What to notice
  • Line 16 pins Flower to the version used for the deployment runtime demo.
  • Line 17 keeps the model in PyTorch so FedAvg can average tensors directly.
  • Lines 20-21 provide preprocessing and parquet IO.
  • Line 22 keeps the dataset download reproducible through Kaggle.
  • Lines 28-33 keep notebook rendering dependencies in the docs extra instead of the Flower runtime images.
