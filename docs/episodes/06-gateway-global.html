06 - Regional gateway and global aggregation

The key piece: the global layer sees regions as clients, not hospitals.

ClientApp in gateway mode

The gateway does not train. It loads the already aggregated regional checkpoint and its metadata. It returns that model to the global SuperLink as a client update.

What to notice
  • Lines 87-92 state the key design contract: the gateway does not train on rows.
  • Lines 94-100 compute the expected regional checkpoint path from region and global-round.
  • Lines 102-110 fail if either the checkpoint or metadata is missing. This prevents the global layer from silently aggregating stale data.
  • Lines 112-119 load the regional model and expose num-examples plus regional validation metrics.
  • Lines 122-130 return the regional model as a normal Flower client update.

ServerApp global

The global layer uses FedAvg again, but now its clients are gateways. Each gateway is weighted by the total number of training examples in its region.

What to notice
  • Lines 124-131 configure FedAvg at the global level. Here the clients are gateway SuperNodes, not hospitals.
  • Lines 133-142 run exactly one global aggregation round for each outer loop iteration.
  • Lines 144-155 save the global checkpoint and metadata after the gateway updates are aggregated.
  • Line 152 records the global training-example total, which should match the sum of regional totals.
