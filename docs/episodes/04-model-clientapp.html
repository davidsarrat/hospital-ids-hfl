04 - Model and site ClientApp

How local network-flow parquet is loaded, the MLP is trained, and an update is returned to Flower.

Dataset and MLP

The main model is intentionally simple so FedAvg remains transparent: a tabular MLP with one logit output and BCEWithLogitsLoss.

What to notice
  • Lines 12-30 load one parquet split and validate that all features are finite.
  • Lines 24-25 separate labels from features and convert them to float32 arrays for PyTorch.
  • Lines 42-63 define the MLP. The final layer has one output because this is binary benign-vs-attack classification.
  • The model is intentionally averageable: its state dict is a set of tensors that Flower can aggregate with FedAvg.

Local training

What to notice
  • Lines 88-97 compute pos_weight to reduce the impact of class imbalance during binary classification.
  • Lines 110-112 create the weighted BCE loss and AdamW optimizer used by every site.
  • Lines 117-129 run the local epochs. No server or other site's network-flow rows are accessed here.
  • This function knows nothing about Flower. That makes the ML task testable on its own and keeps Flower-specific logic in the ClientApp.

Site ClientApp

When the node role is hospital, the service is acting as a healthcare-network site. Flower sends initial arrays. The ClientApp loads those weights, trains on /data/train.parquet, evaluates on validation, and returns arrays plus metrics. The num-examples key enables weighted FedAvg.

What to notice
  • Lines 32-41 route by node role. Sites train; gateways return regional checkpoints.
  • Lines 44-55 load only the mounted site data directory and the arrays sent by Flower.
  • Lines 56-64 execute local training and validation.
  • Lines 66-80 return two records: arrays for model parameters and metrics for aggregation metadata.
  • Line 68 is critical: Flower's FedAvg uses num-examples as the weight for this site update.
